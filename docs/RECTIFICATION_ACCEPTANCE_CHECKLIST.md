## 1. 使用说明

本文件用于做“是否完成整改”的验收，不用于描述整改思路。

使用方式：

1. 执行 AI 完成任务后先自检
2. 审查 AI 按清单逐项复检
3. 联调 AI 对 P0 / P1 主链路做契约复核
4. 项目负责人根据勾选情况决定是否进入下一阶段

规则：
- 关键项未通过，不允许进入下一阶段
- 每一项必须能被代码、接口或运行结果证明
- 不允许靠口头说明视为通过

## 2. P0 联调恢复验收

### P0-1 API 路径统一

- [ ] 前端业务请求路径已统一使用 `/v1/*`
- [ ] 前端源码中不再出现旧路径 `/auth/*` 作为业务请求
- [ ] 前端源码中不再出现 `/api/*` 作为业务请求字面量（不含 `BASE_URL`）
- [ ] 前端源码中不再混用 `/users`、`/agents`、`/roles`、`/permissions` 等无版本前缀路径
- [ ] 前端 API 路径与后端 Controller 路径逐一对齐

### P0-2 响应结构统一

- [ ] `request.ts` 只返回一种成功响应结构
- [ ] 业务层统一按 `res.code` / `res.message` / `res.data` 使用
- [ ] 主链路中不再出现 `res.data.data`
- [ ] 主链路中不再依赖 `AxiosResponse` 结构猜测响应内容
- [ ] `as any` 不再被用于主链路响应读取

### P0-3 token / refreshToken / userInfo 读取统一

- [ ] `token` 读取逻辑统一
- [ ] `refreshToken` 读取逻辑统一
- [ ] `userInfo` 读取逻辑统一
- [ ] `localStorage` / `sessionStorage` 双存储读取策略一致
- [ ] 登出后 localStorage 与 sessionStorage 均被清理
- [ ] “记住我=false” 时后续请求仍可带上登录态

### P0-4 SSE 契约恢复

- [ ] `streamChat` 前后端请求方法一致
- [ ] `streamAgentExecution` 前后端请求方法一致
- [ ] SSE 路径与后端控制器路径一致
- [ ] SSE 请求头包含正确的 `Authorization`
- [ ] 调试页可收到 SSE 数据流
- [ ] 不再出现 SSE 相关 404 / 405

### P0-5 旧路径残留清理

- [ ] `WebMvcConfig` 中旧认证路径残留已清理
- [ ] `SecurityConfig` 中放行路径与真实控制器路径一致
- [ ] 不再存在 `/auth/**` 与 `/v1/auth/**` 混用
- [ ] 旧路径若保留兼容，有明确注释说明

## 11. 最终判定规则

### 允许进入下一阶段的条件

必须同时满足：
1. 当前阶段关键项全部通过
2. 不存在 P0 回归
3. 不存在已确认但未修复的主链路断裂
4. 有清晰的剩余问题说明

### 不允许进入下一阶段的条件

任一满足即不得进入下一阶段：
- 登录主链路仍断裂
- 响应结构仍混乱
- 路径仍未统一
- SSE 仍不可用
- token 存储读取策略仍不一致
- 新回归未处理
